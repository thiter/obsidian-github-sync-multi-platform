# GitHub API 政策与实现指南

本文档总结了本项目中使用的 GitHub API 政策、限制以及具体的实现细节。旨在帮助用户理解我们为什么选择这种方案，以及在使用过程中需要注意什么。

---

## 1. 核心认证政策
我们所有操作都使用 **GitHub 个人访问令牌 (PAT)**。

*   **优点：**
    *   **完全控制：** 用户拥有自己的数据。插件不会将您的笔记存储在任何第三方服务器上。
    *   **支持私有库：** 安全地同步到私有仓库，确保您的笔记内容保密。
    *   **精细权限：** 您只需授予 `repo` 范围（或针对单个仓库的细粒度访问权限）即可。
*   **缺点：**
    *   **手动设置：** 用户必须在 GitHub 上手动生成令牌。
    *   **安全责任：** 令牌实际上就是密码，用户必须妥善保管。

## 2. 速率限制（“交通规则”）
GitHub 对每小时可以发出的请求数量有明确限制。

*   **政策：** 使用 PAT 进行的身份验证请求限制为 **每小时 5,000 次**。
*   **优点：**
    *   **笔记同步绰绰有余：** 即使是每小时写 100 篇笔记的高频用户，也只会消耗这一限制的一小部分。
    *   **实时感：** 允许插件在您停止输入后几乎立即同步。
*   **缺点：**
    *   **批量操作：** 如果不加管理，第一次同步拥有数千个文件的巨大库时可能会达到限制。
    *   **限制共享：** 使用相同 PAT 的其他 GitHub 相关工具会共享这 5,000 次请求的额度。

## 3. 文件大小与类型限制
GitHub REST API 具有特定的物理限制。

*   **API 限制：** 用于同步的 `contents` API 支持高达 **100 MB** 的文件。但是，大于 **1 MB** 的文件需要特殊处理（通常需要底层 Git Data API）。
*   **项目策略：** 我们针对 Markdown 文本进行了优化。对于大型资源（图片/PDF），我们将它们作为二进制数据块处理。
*   **优点：**
    *   **多功能性：** 除 Markdown 外，还支持图片、PDF 和 Canvas 文件。
*   **缺点：**
    *   **二进制开销：** 二进制文件的每次更新都需要重新上传整个文件，对于非常大的资源可能较慢。
    *   **仓库膨胀：** 在 Git 中存储大量大型二进制文件会使仓库随时间显著增大。

## 4. 并发与冲突解决
我们如何处理在两个设备上同时编辑同一个文件的情况。

*   **政策：** 我们使用 **基于 SHA 的验证**。GitHub 要求在允许更新之前提供文件的“当前”SHA 值。
*   **优点：**
    *   **数据完整性：** 防止“盲目覆盖”。如果 GitHub 发现您提供的 SHA 已过期，它会拒绝更改，从而提醒插件存在冲突。
*   **缺点：**
    *   **冲突逻辑：** 需要插件决定是合并、覆盖还是创建副本（通常采用“最后写入者胜”或“冲突副本”策略）。

## 5. 隐私与数据存储
*   **政策：** 所有数据直接在 **Obsidian 应用程序** 和 **GitHub 服务器** 之间传输。
*   **优点：**
    *   **无中间件：** 没有可能被黑客攻击或下线的中间“同步服务器”。
    *   **合规性：** 非常适合对数据隐私有严格要求的用户。
*   **缺点：**
    *   **依赖 GitHub：** 如果 GitHub 的 API 宕机，同步也会随之停止。

1.  **“数据归你所有”：** 强调 GitHub 仓库是数据的唯一真相来源，由用户完全掌控。
2.  **“5000 次俱乐部”：** 告诉用户每小时 5000 次的额度对于记笔记来说非常慷慨，无需担心。
3.  **“默认私有化”：** 展示如何通过私有仓库 + PAT 确保笔记不被外界窥视。
4.  **“不止是文字”：** 简要提到对图片的支持，以及 API 如何处理非文本文件。
5.  **“安全第一”：** 解释 SHA 校验如何确保您在多设备同步时不会意外丢失工作内容。
